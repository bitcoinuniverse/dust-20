Safety and operations

Make the irreversible step boring.

DUST-20's colored-UTXO model makes output review central to user safety. The transaction can confirm while the intended allocation is wrong. Good tooling prevents that mismatch before a wallet signature is requested.

!

Never ask for a seed phrase or wallet private key

A wallet should sign a fully reviewed PSBT locally. If a product generates, stores, uploads, or transmits a temporary private key, it has crossed a major custody boundary and must be treated as test-only until independently redesigned and reviewed.

Trust boundaries

Every boundary needs a visible control.

Wallets, local builders, remote indexers, listings, and Bitcoin nodes can disagree or fail. The UI should make both the data source and the stopping rule clear.

BROWSER TO WALLET

Sign only reviewed outputs

Render an app-side summary of inputs, output roles, values, network, and fee. The wallet confirmation screen should corroborate, not replace, that review.

APP TO BUILDER

Construct, do not custody

A server may prepare a PSBT, but it must never need the user's seed phrase or signing key. Validate request values and redact logs.

APP TO INDEXER

Observe, do not blindly trust

Record source and freshness. A missing, malformed, stale, or contradictory response blocks a risky action instead of becoming a zero balance.

PSBT review

Show the user the allocation map.

Before opening the wallet, present a stable review panel. It should permit a user or support operator to compare the intended outcome with the decoded PSBT later.

Threat model

Threats that change a user's outcome.

These are not theoretical formatting concerns. Each one can result in a signed transaction, a misleading balance, or a custody failure if it is not detected early.

PSBT SUBSTITUTION

Recipient or change is altered

Decode and compare every output role before prompting the wallet. Reject any response that cannot be explained by the review model.

STALE LISTING

Output was already spent

Revalidate outpoint and listing state immediately before construction. A local marketplace record is not chain truth.

INDEXER DISAGREEMENT

Two sources tell different stories

Do not choose the friendlier result. Show a source conflict, block mutation, and retain redacted evidence for investigation.

COLORED FEE LEAK

Allocation pays miners

Assert conservation of colored backing and source fee funding from cardinal UTXOs. Treat unexplained satoshi differences as a failure.

KEY EXPOSURE

Private material crosses a boundary

Never log or request a seed phrase. Disable the affected flow if private key material could reach a service or telemetry system.

RETRY CONFUSION

Ambiguous broadcast becomes a duplicate action

Retain transaction ID and known status. Disable blind repeat submission until the transaction is inspected.

Incident posture

Stop the risky action, preserve the useful facts.

Bitcoin transactions cannot be reversed through an application rollback. Incident response should focus on disabling the unsafe path, protecting users, and preserving redacted evidence.

STOP

Disable the narrowest affected mutation

Pause buy, transfer, split, merge, or key-sensitive flow. Keep read-only state visibly degraded rather than pretending the system is healthy.

PRESERVE

Capture redacted context

Keep transaction IDs, outpoints, route, build version, source timestamps, and error classification. Do not collect seeds, private keys, or unrelated wallet history.

EXPLAIN

Tell users what is paused and what is known

State whether a transaction may be pending and how to inspect a public transaction ID. Never advise a blind retry.

RELEASE

Re-enable only with a regression test

Document root cause, add a fixture, independently retest the path, and re-enable gradually with the correct owners' approval.

Release gates

What must be true before wider exposure.

Documentation alone cannot make a transaction flow safe. These gates turn the field guide into a concrete readiness checklist.

  • Deploy and mint conformance fixtures cover valid and invalid field combinations, exact arithmetic, and actual inscription output values.
  • PSBT fixture tests assert selected inputs, recipient outputs, colored change, cardinal fee funding, ordinary change, and marketplace charges.
  • Wallet compatibility is recorded on controlled testnet fixtures for every path shown to users.
  • Route ownership, indexer outage, stale listing, replacement transaction, and reorg behavior are tested and documented.
  • Any private-key-handling workflow is removed or security-reviewed before it is made broadly available.
  • Operators rehearse the stop, diagnose, communicate, and re-enable sequence with redacted evidence.
