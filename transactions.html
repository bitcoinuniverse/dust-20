Transactions

Where the colored sats actually go.

DUST-20's distinct idea is that token allocation is carried by Bitcoin outputs. That makes transaction construction the critical state transition. The legacy guide shows a split, but it does not publish a full transfer standard.

InvariantSplitMergeFeesUnknowns

Conservation first

Colored backing is not spare change.

The safest compatibility rule is simple: a normal transfer conserves allocation backing. Use a separate cardinal input to fund miners and show colored change whenever the selected colored amount exceeds the amount sent.

sum(colored inputs) = sum(recipient colored outputs) + sum(colored change)For a normal transfer profile, the right side contains no miner fee deduction. A burn model is not defined by the legacy guide.

Worked split

91 units become 10 units plus an explicit 81-unit remainder.

This is the documented 546-sat unit example. It shows why colored change is transaction-critical: omitting it changes where the backing ends up.

Selected colored input49,686 sats91 units * 546 sats
→
Recipient allocation5,460 sats10 units * 546 sats
Colored change44,226 sats81 units * 546 sats
SELECT

Pick colored inputs deliberately

Do not mix allocation UTXOs into ordinary fee selection. Capture their ticker, unit size, amount, outpoint, and source before building.

CALCULATE

Derive every colored output

Compute receiver allocation and colored change from exact integer units. Every claimed colored output should be divisible by its resolved `unit_sats`.

REVIEW

Name outputs by role

Show recipient allocation, colored change, seller payment, cardinal change, fee funding, and tip as different rows. A raw address list is not enough.

Merge and whole-output moves

Do less when the protocol says less.

Merge and whole-UTXO movement are natural UTXO operations, but the legacy guide does not define a universal interpretation for every input combination. A conservative client verifies compatible identity before it sums anything.

MERGE CANDIDATE

Same resolved deployment only

A conservative merge accepts colored inputs only when they resolve to the same accepted ticker and unit ratio. Its resulting colored output equals the exact sum of their backing sats.

WHOLE-UTXO MOVE

No colored change when no remainder exists

If a user sends every selected colored sat to the recipient, no colored change is needed. The review must still show that the amount equals the entire selected allocation set.

Fee separation

Pay the miner from cardinal sats.

Network fees are ordinary Bitcoin value movement. Under the colored-UTXO model they must not be silently taken from an output that an indexer or recipient expects to represent a token allocation.

SAFE SHAPE

Separate cardinal funding input

Add enough non-colored sats to pay the fee, then send ordinary change to its intended address. The colored input and colored outputs keep their allocation arithmetic intact.

UNSAFE SHAPE

Fee shaved from colored backing

A 49,686-sat colored input that produces only 49,000 sats of claimed colored outputs has a 686-sat unexplained gap. Do not call it a valid allocation transfer simply because the Bitcoin transaction confirms.

!

A missing colored change output is a transfer-critical failure

It is not a formatting issue. It can make the remaining backing untracked, redirected, or interpreted differently by another wallet or indexer. Reject the PSBT before signing.

Unresolved transaction rules

Do not write consensus by accident.

The legacy material does not define enough transfer semantics for independent clients to infer every outcome. These questions need a published indexer and wallet policy.
