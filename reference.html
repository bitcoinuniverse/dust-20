Protocol reference

Fields that change value.

This is a compatibility profile for the DUST-20 material published in the legacy guide. It makes the visible arithmetic explicit, flags the current Universe implementation split, and refuses to invent a consensus transfer format where none has been published.

ProfileDeployMintFieldsValidationVectors
i

Read the profile before copying a payload

The legacy guide presents `p`, `op`, `tick`, `supply`, `unit_sats`, and `max_sats` for deploy, with optional `lim_sats`. It presents `p`, `op`, `tick`, `amt`, and `sats` for mint. It also requires the mint inscription output value to equal `sats`. Treat 546 sats as a documented common unit, not an implicit protocol default.

Deploy payload

Declare the ratio and total backing together.

`max_sats` is derived, not a second independent supply choice. A compatible parser should reject a mismatch instead of silently trusting one of the two values.

Legacy-profile deploy
{
  "p": "dust-20",
  "op": "deploy",
  "tick": "DUST",
  "supply": "1000",
  "unit_sats": "546",
  "max_sats": "546000",
  "lim_sats": "49686"
}

`lim_sats` is optional. It caps satoshi backing in one mint when the relevant indexer applies that rule.

max_sats = supply * unit_satsExample: 1,000 units * 546 sats per unit = 546,000 maximum backing sats.
p / op

Identify the protocol and operation.

The legacy profile uses literal `dust-20` and `deploy` values.

tick

Keep the identifier inspectable.

The legacy guide says tick is not restricted to four bytes. It does not settle case normalization or duplicate deployment resolution.

integers

Use exact integer values.

Do not use decimal or floating-point arithmetic for supply, unit size, limits, or backing calculations.

Mint payload

Make the stated and actual output values agree.

A mint is not safe to classify from JSON alone. The payload, the deployment ratio, the inscription output, and the prior minted total must reconcile.

Legacy-profile mint
{
  "p": "dust-20",
  "op": "mint",
  "tick": "DUST",
  "amt": "91",
  "sats": "49686"
}

The actual inscription output value, sometimes shown as Sats In Item, must also be 49,686 sats in this example.

sats = amt * unit_satsExample: 91 units * 546 sats per unit = 49,686 sats.
  • Resolve exactly one accepted deployment before calculating the mint.
  • Require `sats` to equal `amt * unit_sats` with exact integer arithmetic.
  • Require the inscription output value to equal the declared `sats`.
  • When `lim_sats` exists, require `sats <= lim_sats`.
  • Require total prior mints plus `amt` to stay within declared supply.

Compatibility boundary

Do not hide the implementation split.

Current Universe Inscribe behavior is useful evidence, but it should not be described as universal DUST-20 consensus until the profiles are reconciled and independently implemented.

LEGACY PROFILE

Deploy includes `max_sats`

The legacy guide includes `max_sats` and frames it as the product of supply and unit size. Its mint profile includes `sats` and requires matching output value. This site uses that explicit profile for the field reference.

UNIVERSE IMPLEMENTATION

Current builder behavior differs

The current Inscribe builder has been observed to emit deploy payloads with `supply`, `unit_sats`, and `lim_sats`, but without `max_sats`. It also emits a transfer preview with `amt`. Those choices are implementation-specific and need a published compatibility decision.

!

Never downgrade a mismatch to formatting

If an indexer, wallet, or app chooses a different field set or output mapping, show a compatibility warning and stop the irreversible step. Do not silently fill in fields, round satoshi values, or assume an unspecified transfer payload is canonical.
