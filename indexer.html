For app and indexer builders

A balance screen is a helpful view, not the final authority.

Most people can skip this page. It is for people building DUST-20 tools. A balance service reads Bitcoin activity, applies rules, and should clearly say where its number came from and when it might change.

Sources

Tell people where a balance number came from.

A number can come from Bitcoin itself, a balance service, a marketplace, or a cached app response. These are useful for different jobs and should never be silently swapped.

CHAIN

Transaction and outpoint

The durable Bitcoin objects. An indexer must preserve the transaction ID, output index, value, confirmation state, and chain position it observed.

INDEXER

Allocation interpretation

The indexer's determination that a particular outpoint represents a ticker and number of units under its rules.

MARKETPLACE

Listing state

A listing may be useful discovery data, but it is not proof that an output remains unspent or allocation-valid.

CACHE

Application response

A cached response needs source URL, retrieval time, block height or mempool status, and a safe invalidation path.

Allocation record

Store enough context to explain a balance.

A balance is the sum of allocation records. Each record should answer which output, which rule, which source, and what chain state produced its interpretation.

Illustrative indexer allocation record
{
  "outpoint": "txid:vout",
  "tick": "DUST",
  "amount": "91",
  "allocation_sats": "49686",
  "unit_sats": "546",
  "status": "confirmed",
  "source": "indexer",
  "observed_height": 0
}

This is an implementation-facing record shape, not a standardized public API. The key requirement is traceability: a client should be able to inspect how the amount was derived.

Event lifecycle

Index a state transition, not a page load.

The sequence below describes a conservative indexer approach. A production implementation should publish its precise ordering, confirmation, duplicate-deploy, and validation policies.

OBSERVE

Ingest raw chain event

Record transaction, output, inscription association, mempool or block position, and raw payload without prematurely declaring a balance.

NORMALIZE

Parse and validate fields

Apply exact integer parsing and field rules. Keep parse failures visible as invalid events rather than dropping them into an unexplained empty result.

RESOLVE

Resolve deployment and history

Apply the published ticker, duplicate-deployment, mint-limit, and supply policy. Do not guess when two candidates conflict.

MATERIALIZE

Create or update allocation records

Mark the relevant output pending or confirmed with provenance. A spend transitions prior outputs out of spendable state.

RECONCILE

Roll forward or roll back

Update balances after confirmation, replacement, reorg, and spent-output observation. Preserve enough history to explain the transition.

Chain instability

RBF and reorgs are product states.

The legacy guide does not define confirmation or replacement semantics. An indexer must publish them because they directly affect what a wallet may safely spend or list.

RBF OR DOUBLE-SPEND

Invalidate the old observation

If a mempool transaction is replaced or an outpoint is double-spent, mark affected pending allocations and listings invalid or uncertain. Do not retain the old balance as spendable.

REORG

Restore the prior chain view

A reorg can remove a confirmation or replace an observed history. Track block association so affected records can move back to pending, be recomputed, or be invalidated.

i

Show freshness in the UI

A balance response should identify confirmed height, pending state, source, and refresh time. An empty result after a timeout should be shown as unavailable, not as a verified zero balance.

API expectations

Version the response shape or expose the uncertainty.

No canonical DUST-20 indexer API is defined in the legacy guide. Consumers should treat remote responses as a contract that needs schema validation and explicit source policy.

READS

Return provenance

Balance and ticker responses should include a source, observed time or height, pagination, and a state label. Avoid bare arrays that cannot say whether they are complete.

FAILURES

Use a safe error contract

Differentiate unavailable, malformed, stale, invalid-request, not-found, and contradictory-state outcomes. A client must not interpret each as an empty balance.

MUTATIONS

Revalidate near signing

Any UTXO filter, listing validation, or transaction-build request should recheck selected outpoints and return a current timestamp or height with its decision.
