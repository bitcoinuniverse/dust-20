How DUST-20 works

Start safely

Check the basics before you make a transaction.

Good DUST-20 tools show the important details before your wallet opens. You should not have to guess what a generic “send” button will do.

NETWORK

Confirm where you are

Check the Bitcoin network, the wallet, the receiver address, and the fee. Stop if the network or address looks wrong.

BALANCE VIEW

Refresh what the app sees

Ask the DUST-20 balance service for the latest information. It is helpful, but it is still one service’s view of the chain.

WHOLE NUMBERS

Do not round

Use whole units and whole sats only. The app should stop before creating a transaction with a blank, negative, fractional, or rounded amount.

Set the rule

Make the backing rule easy for anyone to check.

A deploy says how many sats back one unit and how many units can exist. The rule should be clear without relying on a hidden app default.

  1. Choose a nonempty ticker and show its exact spelling. The legacy guide does not settle how duplicate names or uppercase and lowercase names should be handled.
  2. Use positive whole-number `supply` and `unit_sats` values. The total backing is `max_sats = supply * unit_sats`.
  3. Show `lim_sats` separately when it is used. It can limit the Bitcoin backing of one mint under the relevant policy.
  4. Show the reviewed values before creating the deploy text. Do not quietly add or change a value while preparing the JSON.
  5. After broadcast, save the transaction ID and wait for confirmation. A deploy does not by itself prove that nobody else used the same ticker.

Create units

Make the token amount and Bitcoin amount match.

For a mint, three numbers must agree: the sats per unit set by the deploy, the `sats` written in the JSON, and the actual Bitcoin output value.

mint_sats = amt * unit_satsWith 91 units and 546 sats per unit, both the payload and inscription output must be 49,686 sats.

What to see before you approve

Show the ticker, amount, sats per unit, total sats, remaining supply, fee, receiver, and the Bitcoin outputs that will pay for the transaction in one clear review screen.

  • Resolve an accepted deployment before accepting `amt`.
  • Reject if `sats != amt * unit_sats` or if actual output value differs from `sats`.
  • Reject if `sats > lim_sats` where a deployment or indexer has a limit.
  • Reject if prior minted units plus `amt` exceed supply.
  • Wait for confirmation and reconciliation before presenting the new output as an available balance.

Send units

When you send part, put the rest in change.

If you send 10 units from an output holding 91, the remaining 81 need their own DUST-20 output. This is called DUST-20 change.

DUST-20 SIDE

Keep the backing balanced

The sats you start with should equal the sats sent to the receiver plus the sats in your DUST-20 change. The amount must stay divisible by the sats-per-unit rule.

REGULAR BITCOIN

Pay the miner fee separately

Use regular bitcoin for the network fee and ordinary change. Do not take sats from DUST-20 backing to pay a fee.

When to stop

If something is unclear, stop.

A missing balance, uncertain broadcast, or changed output is not a successful send. Keep the details, refresh carefully, or ask for help.
